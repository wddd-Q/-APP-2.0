# ADR-002: 数据层架构选择

## 状态
已采纳 (2026-05-27)

## 背景
需要决定游戏数据模型的实现方式。候选方案：Godot Resource 子类、自定义 RefCounted 类、纯 Dictionary。

## 决策
**使用 Godot Resource 子类作为数据模型，保持纯数据层（不发射信号，不含业务逻辑）。**

## 评估标准

| 方案 | 类型安全 | IDE补全 | 序列化 | 编辑器可见 |
|------|----------|---------|--------|-----------|
| Resource 子类 | ★★★★★ | ★★★★★ | ★★★★★ | ★★★★★ |
| RefCounted | ★★★★☆ | ★★★★☆ | ★★☆☆☆ | ★★★☆☆ |
| Dictionary | ★☆☆☆☆ | ★☆☆☆☆ | ★★★★★ | ★★☆☆☆ |

## 理由

1. **类型安全**: @export 标记 + 类型提示让错误在编辑期暴露
2. **编辑器集成**: .tres 文件可在 Godot 编辑器中直接查看和调试
3. **序列化支持**: Resource 有内置序列化，方便存档和网络传输
4. **纯数据约束**: 强制分离数据与逻辑，控制器层处理所有业务逻辑

## 后果

- 正向: 数据模型可以在编辑器中手动创建测试数据
- 正向: 新系统只需新建 Resource 子类文件
- 负向: Resource 不能使用 constructor 参数，需用工厂方法
- 约束: Resource 子类中禁止调用 EventBus、禁止业务逻辑

## 替代方案

**RefCounted**: 放弃了。虽然更轻量，但缺少编辑器集成和内置序列化。

**纯 Dictionary**: 放弃了。虽然最灵活，但类型安全为零，随着系统增多维护成本指数增长。
